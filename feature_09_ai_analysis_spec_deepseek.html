01 Feature definition & scope

Every proxy submission triggers an AI analysis. The analysis produces a structured report visible only to the diaspora owner. The report is always generated — even with no reference material — and becomes progressively richer as more context is provided.

The AI receives: (a) the proxy's submitted photos, (b) the current milestone label, (c) any reference material the owner has uploaded (drawings, renders, reference photos), and (d) the project's phase history. It returns a structured JSON object that the app renders as a human-readable report card on the owner's dashboard.

IN SCOPE — MVP
Photo scene description
Milestone stage assessment
Anomaly detection
Reference comparison (when available)
Confidence scoring
AI limitation disclosure
Owner-only report card
OUT OF SCOPE — MVP
Structural safety assessment
Quantity surveying / cost estimation
Regulatory compliance checking
Real-time video analysis
AI-to-proxy direct messaging
Automated fund release decisions
02 Analysis pipeline
STEP 01 — PROXY ACTION
Proxy submits photos at site
1–10 photos uploaded via two-tap flow. Each photo is geotagged and timestamped by the app before upload. Photos are compressed to ≤1.5MB each for bandwidth efficiency while preserving enough detail for visual analysis. Raw originals stored separately.
STEP 02 — CONTEXT ASSEMBLY
Backend assembles analysis payload
The server collects everything the AI needs: compressed submission photos, current milestone label and description, all reference materials the owner has uploaded (drawings, renders, reference photos — each tagged by type), and the last 3 submission summaries for continuity context.
payload = { milestone: "Roofing — timber frame", submission_photos: [img_1, img_2, img_3], reference_files: [{ type: "floor_plan", file: pdf_raster }, { type: "3d_render", file: render_jpg }], phase_history: ["Foundation ✓", "Block-laying ✓", "Lintel ✓"], prior_summaries: ["Week 12: walls complete...", ...] }
STEP 03 — PRE-PROCESSING
Reference materials prepared for vision input
PDF drawings are rasterised to high-resolution JPEGs (300 DPI) page by page. The relevant page (typically ground floor plan or structural drawing) is selected based on the current milestone. 3D renders are passed as-is. Floor plans have their scale stripped — the AI is told the drawing type but not asked to interpret dimensions precisely.
STEP 04 — AI ANALYSIS
DeepSeek vision API called with full context
A single API call sends all images (submission + references) with the structured system prompt. The model is instructed to return only valid JSON matching the report schema. Max tokens set to 1800. Temperature 0 for consistency. The call is async — the owner sees "Analysis in progress" for 15–45 seconds.
STEP 05 — VALIDATION
Response validated and sanitised
JSON is parsed and validated against the schema. Each field is type-checked. Confidence scores are clamped to valid ranges. If parsing fails (rare model error), a fallback report is generated from a simpler second call asking for plain text only, which is then wrapped in the schema structure. The submission is never left without a report.
STEP 06 — DELIVERY
Report stored, owner notified
Validated report JSON is stored against the submission record. Owner receives a push notification: "AI report ready for [Milestone name]." The report is rendered on-demand from the stored JSON — no re-calling the AI. Reports are immutable once stored; any re-analysis (e.g. after owner uploads new reference material) creates a new versioned report, not an overwrite.
03 Reference input types & handling

The system accepts four reference types. Each unlocks a different depth of comparison analysis. Reference materials are uploaded once by the owner during project setup and remain attached to the project throughout.

TYPE A Floor plan / architectural drawing (PDF or image) Richest comparison

What the AI can do: Identify whether visible rooms, openings, wall arrangements, and structural elements correspond roughly to the drawn layout. Detect obvious deviations (e.g. a wall where none is drawn, a missing opening).

What it cannot do: Read dimensions accurately. Verify setbacks, angles, or structural specifications. Assess concrete mixes, rebar spacing, or load-bearing calculations.

Pre-processing: PDF rasterised at 300 DPI. First relevant page auto-selected based on milestone (ground floor for early phases, roof plan for roofing). Owner can pin the correct page manually.

TYPE B 3D architectural render Strong visual match

What the AI can do: Compare visible external features — roof shape, window positions, facade materials, overall massing — against the render. Most effective during finishing and external work phases.

What it cannot do: Compare internal work or structural elements not visible in the render. Assess colour accuracy in variable lighting conditions.

Pre-processing: Passed as-is. Multiple render angles accepted (max 4). Owner labels each angle (front, rear, left side, right side) during upload.

TYPE C Reference photographs (similar completed build) Useful baseline

What the AI can do: Compare workmanship quality, material appearance, and construction stage against a known-good example. Flag obvious quality discrepancies (rough versus smooth plaster, aligned versus misaligned blocks).

What it cannot do: Make structural comparisons. The reference is aesthetic and qualitative only.

Pre-processing: Passed as-is. Owner labels each photo with what it shows (e.g. "finished plastering — external wall"). Max 6 reference photos.

TYPE D No reference uploaded Description only

What the AI does: Generates a full description-only report: what is visible, what construction activity appears to be in progress, anomalies or concerns, and what cannot be determined. The report is still structured and valuable — it is simply not a comparison.

When this applies: Owner has not uploaded any reference material. Also used as the fallback if reference files fail to process.

04 Report schema — all 6 sections

The AI returns a single JSON object. All six sections are always present. Sections 5 and 6 are populated only when reference material exists; they contain a not_applicable value otherwise.

S1Scene descriptionAlways present

A plain-language description of what is visible across all submitted photos. Written for a non-technical owner. Describes the site conditions, visible work, materials present, number of workers (if any), weather/lighting, and any notable site features.

scene_summarystring2–4 sentence overview of site conditions
photos_describedarray[object]Per-photo description: photo_index, what_visible, notable_details
workers_presentboolean | nullNull if indeterminate from photos
materials_visiblearray[string]List of identifiable materials on site
S2Milestone stage assessmentAlways present

The AI's assessment of whether the visible work corresponds to the milestone the proxy has tagged. If the milestone is "Roofing — timber frame" and the photos show completed block walls but no timber, this section flags the discrepancy without accusing — it describes what it sees and what would be expected.

declared_milestonestringPassed in from proxy submission tag
stage_appears_consistentboolean | nullNull if insufficient evidence
stage_assessment_textstringPlain-language assessment, 2–5 sentences
expected_vs_visiblestringWhat the AI expected to see for this milestone vs what it observed
S3Anomaly & concern detectionAlways present

Things that look unusual, incomplete, or potentially problematic. The AI is calibrated to flag observations, not conclusions — it notes what it sees, not what it means structurally. Every finding is prefaced with an epistemic qualifier (e.g. "appears to show", "may indicate", "worth asking about").

anomaliesarray[object]Each item: description, photo_index, severity (low/medium/high), suggested_question
inactivity_signalbooleanTrue if photos suggest site is not actively worked (no workers, no fresh materials, unchanged vs prior submission)
positive_observationsarray[string]Things that look correct or well-executed
S4AI limitations disclosureAlways present

An explicit, honest list of what could not be assessed from the photos. This section is mandatory and cannot be empty — the AI must always disclose its limitations for this specific submission. It is rendered visibly in the report to calibrate owner trust.

cannot_assessarray[string]Specific things not determinable (e.g. "concrete mix quality", "rebar spacing inside columns")
photo_quality_issuesarray[object]Blurry, too dark, too far: photo_index + issue description
inspector_recommendedbooleanTrue if AI assessment is significantly limited by photo quality or complexity
S5Reference comparisonWhen reference exists

Comparison of what is visible in the submission against the reference material. Each comparison item is tagged with the reference type it uses. The AI never makes absolute judgments — it expresses observations as degrees of correspondence.

reference_types_usedarray[string]Which reference types were available for this comparison
comparison_itemsarray[object]feature_name, reference_expectation, observed, correspondence (matches / partial / differs / cannot_determine)
overall_correspondencestring enumstrong / partial / limited / insufficient_evidence
S6Confidence scoresAlways present

Numeric confidence scores (0.0–1.0) for each major claim the AI makes. Lower scores appear with explicit uncertainty language in the rendered report. Scores below 0.4 trigger a recommendation to dispatch an inspector.

scene_description_confidencefloat 0–1How clearly visible the site conditions are
milestone_assessment_confidencefloat 0–1How confidently the milestone can be assessed
reference_comparison_confidencefloat 0–1 | nullNull if no reference. Quality of comparison possible.
overall_confidencefloat 0–1Weighted composite. Drives the summary badge colour.
05 System prompt design

The system prompt is assembled dynamically per submission. Static sections are cached; variable sections (milestone, reference context, phase history) are injected at runtime.

SYSTEM PROMPT 
## ROLE
You are a construction progress analyst for a diaspora construction
monitoring application. You help Nigerians living abroad track building
projects back home by analysing photos submitted from the site.

## AUDIENCE
Your report is read by the diaspora owner — a non-technical person who
is emotionally invested in this project. Write clearly, avoid jargon,
and never present uncertain observations as facts.

## CONTEXT FOR THIS SUBMISSION
Project:          {project_name}
Current milestone: {milestone_label} — {milestone_description}
Completed phases:  {phase_history_list}
Submission date:   {submission_date}
Reference files:   {reference_summary} // e.g. "floor plan (PDF, ground floor) + 3D render (front elevation)"

## IMAGE ORDER
Images are provided in this order:
  1. Submission photo 1 of {n}
  ... (all submission photos)
  {n+1}. Floor plan — ground floor [IF PROVIDED]
  {n+2}. 3D render — front elevation [IF PROVIDED]
  ... (all reference images, labelled)

## ANALYSIS INSTRUCTIONS

1. SCENE DESCRIPTION
   Describe what you can see across all submission photos. Be specific
   about visible construction elements. Note worker presence, materials,
   and site conditions. Do not describe what you expect — only what is
   visible.

2. MILESTONE ASSESSMENT
   State whether what is visible is consistent with the declared
   milestone. If it appears inconsistent, describe the discrepancy
   factually without implying deception. Use phrases like "the photos
   appear to show" not "the photos prove."

3. ANOMALY DETECTION
   Flag anything unusual: unfinished work at visible edges, materials
   that appear inconsistent with this phase, apparent site inactivity,
   water pooling, structural elements that appear misaligned. For each
   anomaly, generate a plain-language question the owner could ask
   their proxy. Rate severity: low (aesthetic), medium (worth monitoring),
   high (warrants immediate query or inspection).
   ALSO note positive observations — things that look correct.

4. LIMITATIONS
   Be explicit about what cannot be determined from these photos. Always
   include at least 3 items. Never omit this section.

5. REFERENCE COMPARISON [ONLY IF REFERENCE FILES PROVIDED]
   Compare specific visible features against the reference material.
   For each comparison, state what the reference shows, what you observe,
   and your assessment of correspondence. Use: matches / partial / differs
   / cannot_determine. Never guess at dimensions or structural compliance.

6. CONFIDENCE SCORES
   Score each section 0.0–1.0. Consider photo quality, angle coverage,
   lighting, and the complexity of what is being assessed.

## CALIBRATION RULES
- Never use the words "fraud", "theft", "cheating", or "lying"
- Never make structural safety assessments
- Never state that work is definitely complete or definitely not complete
- Always qualify claims about internal/hidden elements (e.g. "the exterior
  of the column appears complete; the internal reinforcement cannot be
  assessed from these photos")
- If a photo is too blurry, dark, or distant to analyse, say so explicitly
  in the photo_quality_issues field

## OUTPUT FORMAT
Return ONLY valid JSON matching the schema below. No preamble, no
explanation, no markdown fences. Your entire response must be parseable
JSON.

{schema_definition}
06 Confidence scoring model

Confidence scores drive visual treatment in the rendered report and determine whether an inspector dispatch prompt is shown. They are AI-generated but clamped and validated server-side.

0.8 – 1.0 High confidence
Clear photos, good coverage, visible elements match expectations. Report rendered with solid colours and direct language.
0.55 – 0.79 Moderate confidence
Some photo quality issues or partial coverage. Report renders with amber accents and hedged language ("appears to show").
0.4 – 0.54 Low confidence
Poor photos, limited angles, or complex milestone. Inspector dispatch prompt shown. Language uses strong qualifiers throughout.
0.0 – 0.39 Very low confidence
Analysis severely limited by photo quality. Description-only report. Inspector dispatch strongly recommended. Owner prompted to request better photos.

The overall_confidence composite is weighted: scene description (0.2) + milestone assessment (0.4) + reference comparison (0.4 if available, else 0). If no reference exists, milestone assessment is weighted 0.6 and scene description 0.4.

08 Edge cases & failure modes
All photos unusable
If every photo is too dark, blurry, or fails to load, the report is generated with confidence 0.1. Section S1 states "No usable images received." Owner prompted: "Ask your proxy to resubmit clearer photos." Submission flagged for re-upload request.
API timeout / model failure
If the DeepSeek API call times out or returns an error, a second simplified call is made (text only, no images). If that also fails, a placeholder report is stored: "Analysis could not be completed for this submission. Photos are saved and visible above." Retried automatically after 30 minutes.
Photos appear off-site
If geotag coordinates are more than 500m from the registered site address, the report includes a flagged note: "These photos were taken approximately X km from the registered site location." The AI still analyses what it sees. The owner decides how to interpret the discrepancy.
Reference file unreadable
If the PDF fails to rasterise or the reference image is corrupt, the report falls back to Type D (no reference) and notes: "Your uploaded floor plan could not be processed for this analysis." The submission is still analysed. Owner is prompted to re-upload the reference file.
Invalid JSON returned
If the model returns malformed JSON (rare), the raw text is stored for debugging. A second call is made with a simpler prompt asking for plain text analysis. This plain text is wrapped in a minimal schema object and stored. Report renders in a "simplified mode" with a notice.
Duplicate submission photos
Server-side perceptual hash check before the API call. If two photos are >95% identical, one is deduplicated and a note added: "1 duplicate photo was excluded from analysis." This prevents inflated confidence from repeated near-identical images.
09 API call structure

A single POST /v1/messages call with vision. All images are base64-encoded. The message array contains the system prompt followed by a single user turn containing all images and the analysis instruction.

JavaScript / Node 
const analysisPayload = {
  model: "deepseek-sonnet-4-20250514",
  max_tokens: 1800,
  temperature: 0,
  system: buildSystemPrompt({
    projectName:        submission.projectName,
    milestone:          submission.milestoneLabel,
    milestoneDesc:      submission.milestoneDescription,
    phaseHistory:       project.completedPhases,
    referenceFileSummary: describeReferenceFiles(project.referenceFiles),
    schemaDefinition:   REPORT_SCHEMA_JSON,
  }),
  messages: [{
    role: "user",
    content: [
      // Submission photos first, labelled
      ...submission.photos.map((photo, i) => ({
        type: "image",
        source: {
          type: "base64",
          media_type: photo.mimeType,
          data: photo.base64,
        },
      })),
      // Reference files after, each with a text label
      ...project.referenceFiles.flatMap((ref, i) => [
        { type: "text", text: `Reference file ${i+1}: ${ref.label}` },
        { type: "image", source: { type: "base64", media_type: ref.mimeType, data: ref.base64 } },
      ]),
      // Final instruction turn
      {
        type: "text",
        text: `Analyse the ${submission.photos.length} submission photo(s) above`
             + (project.referenceFiles.length > 0
               ? ` against the ${project.referenceFiles.length} reference file(s) provided.`
               : `. No reference files are available for this project.`)
             + ` Return only valid JSON matching the schema in your instructions.`,
      },
    ],
  }],
};

const response = await fetch("https://api.anthropic.com/v1/messages", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify(analysisPayload),
});

const raw = await response.json();
const reportJson = JSON.parse(raw.content[0].text);
await validateAndStore(submission.id, reportJson);
10 Constraints & honest limitations
Latency: 15–45 seconds
Vision API calls with multiple large images take time. The report is async — shown as "in progress" on the dashboard. Push notification sent when complete. Never block the proxy submission flow on AI response.
Cost per analysis: ~$0.04–0.12
Estimated based on ~10–15 images at Sonnet pricing. With 4 submissions/month per project, this is ~$0.20–0.50/project/month. Factored into the subscription pricing — not a per-use charge visible to owners.
Cannot assess structural integrity
Rebar spacing, concrete grade, load calculations, footing depth — none of these are visible in photos and the AI is explicitly instructed never to comment on them. This is a fundamental ceiling, not a future feature.
Cannot detect staged photos
If a proxy submits photos taken previously, the AI cannot reliably detect this beyond obvious inconsistencies (different scaffolding, different vegetation state). Geotag verification is the primary defence; the AI is a secondary layer, not the fraud prevention mechanism.
Drawing interpretation is approximate
The AI reads floor plans visually, not as CAD data. It identifies room shapes, opening positions, and general layout — not dimensions. Scale and precise measurements are not assessed. This must be stated clearly in the report and in owner onboarding.
Report always generated
Even with poor photos and no reference material, a description-only report is generated. The owner always receives something useful from every proxy submission, even if it is only "the site appears active and the following materials are visible."
11 What the proxy sees (and does not see)

The proxy sees only a submission confirmation screen. They never see the AI report. This is a deliberate product decision with three reasons: it prevents the proxy feeling surveilled or critiqued, it avoids confusion ("the AI said my photos were blurry"), and it preserves the owner's information advantage when having conversations with the proxy about what they've observed.

PROXY SEES
"Update submitted successfully"
Their own submission history (dates, milestone tags, photo count)
Milestone checklist — what's been submitted, what's still outstanding
If the owner requests better photos — a simple nudge notification with the area to focus on (no AI language, framed as coming from the owner)
PROXY DOES NOT SEE
AI report or any AI-generated language
Confidence scores
Anomaly flags or concern ratings
Reference comparison results
Inspector recommendations
